================================================================
SECTION 1 — THE ONE THING BOTH TEAMS MUST AGREE ON FIRST
================================================================

Before touching a single line of integration code, both teams
need to sit together (or on a call) and confirm these 5 things.
Everything else follows from them.

1. BASE URL
   Where is the backend deployed?
   Example: https://api.yourconsultingfirm.com
   The frontend must store this as a single environment variable.
   NEVER hardcode it. Every environment (dev, staging, prod) will
   have a different URL.

   Frontend .env:
     NEXT_PUBLIC_API_URL=https://api.yourconsultingfirm.com
     or
     VITE_API_URL=https://api.yourconsultingfirm.com

2. CORS
   The backend must whitelist the frontend's exact domain.
   Backend needs to add to app.ts:
     app.use(cors({
       origin: ['https://yourconsultingfirm.com', 'http://localhost:3000'],
       credentials: true,
     }));
   Frontend must send: credentials: 'include' if using cookies,
   or just set Authorization header if using tokens (our approach).

3. TOKEN STORAGE
   We use JWT tokens. Frontend must decide where to store them.
   Recommendation: memory (React state/context) for the token itself,
   with an httpOnly cookie for refresh if you later add refresh tokens.
   DO NOT store JWTs in localStorage — it is vulnerable to XSS.
   Current setup has no refresh tokens — 30 days for user, 7 days for admin.

4. RAZORPAY KEY
   The backend holds RAZORPAY_KEY_SECRET (never exposed to frontend).
   The frontend needs RAZORPAY_KEY_ID (public key, safe to expose).
   The frontend loads the Razorpay JS SDK and uses the KEY_ID.
   Backend returns the orderId — frontend passes it to Razorpay SDK.

5. RESPONSE SHAPE
   Every single response from this backend — success or error — is:
   {
     "success": true | false,
     "message": "Human readable string",
     "data": { ... } | null
   }
   Frontend should never assume shape — always read success first,
   then read data, then message for display.

new_engagement          → Admin notified when any payment succeeds

new_message             → Recipient notified on every message (user↔admin)

================================================================
SECTION 5 — FRONTEND INTEGRATION: STEP BY STEP
================================================================

This is the exact sequence two separate teams should follow
to connect frontend and backend without breaking things.

──────────────────────────────────────────────────────────────
PHASE 1: ENVIRONMENT & CONNECTION (DAY 1)
──────────────────────────────────────────────────────────────

Step 1 — Backend team deploys to a staging server.
  Even a basic VPS or Railway/Render free tier works.
  The backend must be reachable at a real HTTPS URL, not localhost.
  Localhost only works if both teams are on the same machine.

Step 2 — Backend team adds CORS.
  In app.ts, add:
    import cors from 'cors';
    app.use(cors({
      origin: [
        'http://localhost:3000',   // frontend dev
        'https://yourfrontend.com' // frontend production
      ],
      credentials: true,
    }));
  Install: npm install cors && npm install --save-dev @types/cors

Step 3 — Frontend team creates their API base config.
  Create one file: lib/api.ts (or similar)
    const BASE_URL = process.env.NEXT_PUBLIC_API_URL;

    export async function apiRequest(
      method: string,
      path: string,
      body?: object,
      token?: string
    ) {
      const res = await fetch(`${BASE_URL}${path}`, {
        method,
        headers: {
          'Content-Type': 'application/json',
          ...(token && { Authorization: `Bearer ${token}` }),
        },
        body: body ? JSON.stringify(body) : undefined,
      });
      const data = await res.json();
      if (!data.success) throw new Error(data.message);
      return data.data;
    }

  This single function is used for every API call. All error
  handling flows through the thrown Error with the backend message.

Step 4 — Frontend team tests GET /health.
  If they get { success: true } the connection works.
  This confirms: URL correct, CORS correct, server running.

──────────────────────────────────────────────────────────────
PHASE 2: AUTH FLOWS (DAY 1-2)
──────────────────────────────────────────────────────────────

Step 5 — Implement user login.
  POST /auth/login → get token → store in React Context or Zustand.
  DO NOT put in localStorage.
  Token shape: standard JWT — frontend can decode it client-side to
  read userId and expiry without making a network call.

Step 6 — Implement admin login separately.
  POST /admin/login → store admin token separately from user token.
  Admin area and user area should be completely separate flows.

Step 7 — Implement token attachment.
  Every protected request must send: Authorization: Bearer <token>
  Do this once in your api.ts wrapper, not on every individual call.

Step 8 — Implement logout.
  Just delete the token from state. No backend call needed.
  The JWT expires naturally. If you need instant invalidation you
  would add a token blacklist — tell us and we can add that.

──────────────────────────────────────────────────────────────
PHASE 3: PUBLIC PAGES (DAY 2-3)
──────────────────────────────────────────────────────────────

Step 9 — Services/pricing page.
  GET /services → render service cards.
  GET /services/:id → service detail page.
  These are completely public, no token needed.

Step 10 — Coupon field on checkout.
  POST /coupons/validate with { code, serviceId }
  On success: show finalPrice. Store couponId for order creation.
  On error: show the exact message from data.message.

Step 11 — Contact form.
  POST /contact — show success message, clear form.

Step 12 — Newsletter.
  POST /newsletter/subscribe — show confirmation. Handle 409 gracefully
  ("You are already subscribed" rather than showing an error).

──────────────────────────────────────────────────────────────
PHASE 4: PAYMENT FLOW (DAY 3-4)
──────────────────────────────────────────────────────────────

This is the most technically complex part. Do this carefully.

Step 13 — Frontend installs Razorpay.
  In your HTML or Next.js _document: load Razorpay checkout script.
  <script src="https://checkout.razorpay.com/v1/checkout.js"></script>

Step 14 — Checkout flow implementation.
  a. User fills form (service already selected, optional coupon validated)
  b. User clicks Pay → call POST /payments/create-order with user token
  c. Backend returns { orderId, amount, currency, serviceTitle }
  d. Frontend opens Razorpay modal:

    const options = {
      key: process.env.NEXT_PUBLIC_RAZORPAY_KEY_ID,
      order_id: orderId,
      amount: amount,
      currency: currency,
      name: 'Your Consulting Firm',
      description: serviceTitle,
      handler: function(response) {
        // Payment succeeded on Razorpay side
        // Backend webhook has already (or will shortly) fulfill the order
        // Redirect user to their dashboard
        router.push('/dashboard');
      },
      modal: {
        ondismiss: function() {
          // User closed modal without paying — show message
        }
      }
    };
    const rzp = new window.Razorpay(options);
    rzp.open();

  e. On handler callback: redirect to /dashboard.
     Do NOT verify payment client-side. The backend webhook already did it.
     Wait 2-3 seconds or poll GET /engagements until the new engagement appears.

Step 15 — Post-payment state.
  After redirect to dashboard, call GET /engagements.
  The new engagement will be there within 1-2 seconds of payment.
  If it is not there immediately, poll once after 3 seconds.

──────────────────────────────────────────────────────────────
PHASE 5: USER DASHBOARD (DAY 4-6)
──────────────────────────────────────────────────────────────

Step 16 — Dashboard home.
  GET /engagements → list all user engagements.
  Each card: service name, status badge, progress bar (progressPercent).

Step 17 — Individual engagement page.
  GET /engagements/:id → full engagement data.
  Page sections (can be tabs or scrolled):
    - Overview (status, progress)
    - Messages thread
    - Files/deliverables
    - Questionnaires
    - Checklist (read-only progress view)
    - Purchase brief (purchase answers)
    - Feedback (shown only when status === 'delivered')

Step 18 — Messages.
  GET /engagements/:id/messages → render thread.
  POST /engagements/:id/messages → send button.
  Poll GET every 30 seconds OR implement websockets later.
  Show 403 as "Messaging is closed — this engagement has been delivered."

Step 19 — Files.
  GET /engagements/:id/files → list with download links.
  Files are URLs — open in new tab, do not proxy through backend.

Step 20 — Questionnaires.
  GET /engagements/:id/questionnaires → check isSubmitted flag.
  If not submitted: show form with questions, submit button calls POST /questionnaires/:id/submit
  If submitted: show read-only answers.

Step 21 — Notifications.
  GET /notifications on page load and on a 30-second poll.

SECTION 6 — THINGS THAT MIGHT NEED TWEAKING WHEN CONNECTING
================================================================

These are the most common mismatches between separately built
frontend and backend. Check each one.

1. TOKEN HEADER FORMAT
   Backend expects:  Authorization: Bearer eyJhbGci...
   Common frontend mistake: sending just the token without "Bearer "
   or using a different header name like X-Auth-Token.

2. DATE FORMAT
   Backend stores dates as ISO 8601 (2024-03-15T10:00:00.000Z).
   Frontend must send deadline fields in ISO 8601.
   If frontend sends "15/03/2024" the backend will reject it.

3. PRICE / AMOUNT
   Razorpay uses paise (1 rupee = 100 paise) internally.
   The backend stores amount in paise but this is an implementation detail.
   The frontend should display: amount / 100 with ₹ symbol.
   When showing service price from GET /services — this is in rupees as entered.

4. RAZORPAY AMOUNT IN MODAL
   The amount from create-order response is in paise.
   Pass it directly to Razorpay options.amount — do NOT divide by 100 there.
   Razorpay SDK expects paise.

5. COUPON FLOW ORDERING
   Frontend must call POST /coupons/validate BEFORE POST /payments/create-order.
   The couponId from validate must be passed to create-order.
   If frontend skips validate and sends couponCode directly — that also works,
   but the frontend would not know the discounted price before creating the order.

6. ENGAGEMENT STATUS VALUES
   The status field can be: active | delivered
   Frontend must handle both. Do not hardcode checks against other strings.

7. CHECKLIST STEP IDs
   stepId values are strings set by the admin when creating the service.
   They look like "step-1", "step-2" etc. — but the exact format depends on
   what was entered. Frontend should never generate stepIds — only read them.

8. PROGRESS PERCENT
   Returns as a float: 33.333...% not 33%.
   Frontend should Math.round() or toFixed(0) for display.
   But when checking canDeliver use the backend's canDeliver flag — never
   calculate it yourself from the percent.

9. PURCHASE ANSWERS SHAPE
   purchaseAnswers is an array of { question: string, answer: string }.
   The exact questions depend on what was configured per service.
   Frontend should render them generically as key-value pairs.

10. NOTIFICATION RECIPIENT FOR ADMIN
    Admin notifications use recipientId: 'admin-global'.
    This means all admins see the same notifications.
    If you later need per-admin notifications, tell us — it is a model change.

11. FILE UPLOAD
    The backend does NOT handle file uploads.
    Files are URL references only.
    Frontend must upload files to S3, Cloudinary, Google Drive, etc. independently,
    then pass the resulting URL to POST /engagements/:id/files/admin.
    If frontend team assumed the backend handles file upload — this needs discussion.

12. PASSWORD RESET
    Users cannot reset their own passwords.
    There is no "forgot password" email flow.
    Users must contact the organisation who resets it via admin panel.
    If the frontend has a "forgot password" page — it should show a message like
    "Please contact us to reset your password" and link to the contact form.

SECTION 7 — PROFESSIONAL PRACTICES FOR INTEGRATION
================================================================

These are what professional teams do. Not optional extras.

1. USE A SHARED POSTMAN COLLECTION
   Backend team exports the complete Postman collection with all 47 routes,
   example request bodies, and expected responses.
   Frontend team imports it to understand and test endpoints independently.
   Both teams update it when anything changes.

2. NEVER CHANGE A RESPONSE SHAPE WITHOUT TELLING THE OTHER TEAM
   If backend changes data.engagements to data.items — frontend breaks silently.
   Agree: any breaking change to a response shape requires a message to the other team
   the same day, before deploying to staging.

3. VERSION YOUR API IF POSSIBLE
   Mount all routes under /api/v1/...
   This means later you can add /api/v2/... with breaking changes
   while old frontend still works on v1.
   It is a one-line change in app.ts: app.use('/api/v1', router)

4. USE STAGING ENVIRONMENT
   Never test integration on production.
   Backend staging: api-staging.yourconsultingfirm.com
   Frontend staging: staging.yourconsultingfirm.com
   Only promote to production after staging integration is confirmed working.

5. HANDLE EVERY HTTP STATUS CODE ON FRONTEND
   Do not just handle 200. Handle 400, 401, 403, 404, 409, 500 explicitly.
   400 → show validation message to user
   401 → redirect to login
   403 → show "you don't have permission" message
   404 → show not found state
   409 → show "already exists" message
   500 → show generic error, do not expose details to user

6. LOADING AND ERROR STATES ON EVERY API CALL
   Every button that calls an API needs three states:
   - Default (idle)
   - Loading (disable button, show spinner)
   - Error (show message, re-enable button)
   Never let a user click the same button twice during an API call.

7. DO NOT STORE SENSITIVE DATA IN FRONTEND STATE LONGER THAN NEEDED
   Plain passwords from reset-password endpoint: copy to clipboard then discard.
   Never put plain passwords in component state or logs.

8. LOG FRONTEND ERRORS TO A SERVICE
   Use Sentry or similar. When frontend gets a 500 from backend,
   log the full request details. This is how you debug production issues.

9. DOCUMENT EVERY CHANGE DURING INTEGRATION
   Keep a simple INTEGRATION_CHANGES.md file in a shared repo.
   Every time a tweak is made on either side to make them compatible,
   write one line about what changed and why.
   This prevents "why does this work this way" questions three months later.

10. AGREE ON ERROR MESSAGE DISPLAY POLICY
    Backend sends message: "Feedback can only be submitted after the engagement has been delivered."
    Does frontend show this exact string to the user?
    Or does frontend have its own copy?
    Agree on this upfront — inconsistency looks unprofessional to users.
