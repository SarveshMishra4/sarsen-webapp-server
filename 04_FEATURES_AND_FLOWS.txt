FEATURE AREA 8 — PAYMENTS & PURCHASE FLOW
================================================================

FEATURE 8.1 — Create Razorpay Order (Checkout Initiation)
  What it does: Validates purchase details, creates Razorpay order, stores pending record.
  Route: POST /payments/create-order
  Function flow:
    1. requireUser → verifies user JWT, attaches userId
    2. paymentController.createOrder()
    3. createOrderSchema.safeParse() → validates serviceId, userEmail, optional coupon + answers
    4. purchaseFlowService.initiateOrder()
       a. Service.findOne({ _id, isActive: true }) → 404 if not found or disabled
       b. If couponCode provided: couponService.validateCoupon() → same 4-step check
       c. finalPrice = coupon ? coupon.price : service.price
       d. paymentService.createOrder()
          → razorpay.orders.create({ amount: finalPrice * 100, currency: 'INR' })
          → Payment.create() → stores orderId, amount, status: 'pending', purchaseAnswers
       e. Returns { orderId, amount, currency, serviceTitle }
    5. formatResponse() → frontend uses orderId to open Razorpay modal

FEATURE 8.2 — Payment Webhook Fulfillment (Automatic)
  What it does: Verifies payment, creates user + engagement + questionnaire automatically.
  Route: POST /payments/webhook
  Called by Razorpay server — no frontend involvement.
  Function flow:
    1. express.raw() → preserves raw body for HMAC verification
    2. paymentController.handleWebhook()
    3. paymentService.verifyWebhookSignature()
       a. crypto.createHmac('sha256', RAZORPAY_WEBHOOK_SECRET)
       b. hmac.update(rawBody).digest('hex') → compare to Razorpay header
       c. 400 if signature mismatch
    4. purchaseFlowService.fulfillAfterPayment()
       a. Payment.findOne({ razorpayOrderId }) → finds pending record
       b. identityService.resolveOrCreateUser(userEmail) → creates user if new
       c. Engagement.create() → new engagement with:
          - userId, serviceId, couponId
          - engagementChecklist: copied from service.defaultChecklist
          - progressPercent: 0, canDeliver: false, status: 'active'
       d. PurchaseQuestionnaire.create() → stores purchaseAnswers snapshot permanently
       e. Payment.findByIdAndUpdate({ status: 'success', engagementId }) → links records
       f. notificationService.createNotification() × 2
          → type: new_engagement → admin-global
          → type: payment_success → user
    5. Returns 200 always (Razorpay retries on non-200)

================================================================

FEATURE 10.1 — Toggle Checklist Step (Admin)
  What it does: Marks one step complete or incomplete. Progress recalculates instantly.
  Route: PATCH /engagements/admin/:id/checklist/:stepId
  Function flow:
    1. requireAdmin → verifies admin JWT
    2. checklistController.toggleStep()
    3. toggleStepSchema.safeParse() → validates isCompleted is boolean
    4. checklistService.toggleStep()
       a. Engagement.findById() → 404 if not found
       b. engagement.status === 'delivered' → 403 if already delivered
       c. Find step by stepId in engagementChecklist[] → 404 if step not found
       d. step.isCompleted = isCompleted → mutates step
       e. recalculateProgress() → (completedCount / totalCount) * 100 → exact float
       f. engagement.canDeliver = progressPercent === 100 → boolean
       g. engagement.save() → persists changes
    5. formatResponse() → returns updated checklist + progressPercent + canDeliver

FEATURE 10.2 — Replace Entire Checklist (Admin)
  What it does: Restructures the checklist — add, remove, reorder steps.
  Route: PUT /engagements/admin/:id/checklist
  Function flow:
    1. requireAdmin → verifies admin JWT
    2. checklistController.updateChecklist()
    3. updateChecklistSchema.safeParse() → validates steps array (stepId, title, order)
    4. checklistService.updateChecklist()
       a. Engagement.findById() → 404 if not found
       b. engagement.status === 'delivered' → 403 blocked
       c. existingMap = Map of stepId → isCompleted from current checklist
       d. engagement.engagementChecklist = newSteps.map() →
          preserves isCompleted for existing stepIds, defaults false for new ones
       e. recalculateProgress() → recalculates from new structure
       f. engagement.save()
    5. formatResponse() → returns new checklist + updated progress

FEATURE 10.3 — View Checklist and Progress (User or Admin)
  What it does: Returns checklist state, progress %, canDeliver, and status.
  Route: GET /engagements/:id/checklist

       a. Engagement.findById()

    4. formatResponse() → read-only response

FEATURE 10.4 — Deliver Engagement (Admin)
  What it does: Marks engagement complete. Locks messaging. Notifies user.
  Route: PATCH /engagements/admin/:id/deliver
  Function flow:
    1. requireAdmin → verifies admin JWT
    2. checklistController.deliverEngagement()

  Function flow:

    2. notificationController.markAsRead()
    3. notificationService.markAsRead()

          - Admin: { _id, recipientRole: 'admin' }
          - User:  { _id, recipientId: userId, recipientRole: 'user' }

       c. Returns null if not found or wrong owner → 404
