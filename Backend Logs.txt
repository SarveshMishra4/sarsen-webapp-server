========================================================
ARCHITECTURE PRINCIPLE
========================================================

The platform follows a modular capability-based architecture.

Each module represents a distinct capability of the system.

When a new capability is introduced, a new module is created
instead of modifying existing modules. This ensures that
completed modules remain stable and prevents accidental
changes that could break existing functionality.

Modules communicate with each other only through APIs and
shared middleware such as authentication. This design allows
the system to grow incrementally while maintaining stability,
clarity and maintainability.

    ## PROJECT PHILOSOPHY

    This backend system is being built using a modular architecture.
    Each capability of the system is implemented as a separate module.

    A module contains everything required for that capability:

    • routes
    • controller
    • service
    • model
    • validator (if required)

    Modules are independent from each other. Once a module is completed and tested, its internal code should not be modified unless a critical bug is discovered.

    This rule protects the stability of the system and ensures that previously working features do not break when new capabilities are added.

    The architecture is designed so that new functionality is always implemented by creating a new module rather than modifying existing modules.

    ---

    ## TECHNOLOGY DECISIONS

    Database System
    MongoDB

    Database Library
    Mongoose

    We are using Mongoose as the Object Data Modeling (ODM) layer to interact with MongoDB.

    This means:

    • Models define database schemas
    • Models contain database operations
    • Services call model functions
    • Controllers handle HTTP communication
    • Routes expose API endpoints

    We are NOT using Prisma in this project.

    All database operations must be implemented through Mongoose models.

    ---

    ## CORE ARCHITECTURE FLOW

    Each request flows through the following layers:

    Route
    ↓
    Controller
    ↓
    Service
    ↓
    Model (Mongoose)
    ↓
    MongoDB

    Responsibilities of each layer:

    Route
    Defines API endpoints.

    Controller
    Handles request and response objects.

    Service
    Contains business logic.

    Model
    Handles database operations.

    Once a module is completed and tested, it should not be modified.

    New functionality must be implemented as a new module.

    Rule 3
    All database operations must go through Mongoose models.

    Rule 4
    Services must not contain raw database queries.

    Controllers must remain thin and delegate logic to services.

    Routes must only define endpoints and call controllers.

    Every module must follow the same layered structure.
